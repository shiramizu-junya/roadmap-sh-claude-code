## 3. ハンズオンの題材:自作 Notes API(テンプレ・フレームワークなし)

題材は **「メモ(Notes)を管理する小さな Web API」** です。設計方針は3つ。

1. **テンプレート・フレームワークを使わない。** Python の**標準ライブラリ `http.server` だけ**で、HTTP サーバを手書きします。
   → `pip install` も `uv add` も不要。`python` さえあれば動くので、誰でも今すぐ動かせます。
2. **あえて「伸びしろ」を仕込む。** 動くけれど、バグ・未実装・古い書き方が残った状態にします。
   → 後のユースケース(バグ修正・リファクタ・テスト)で**直す対象**になります。
3. **層を分ける。** 「データを持つ箱(`NotesStore`)」と「HTTP の入出力(`Handler`)」を分けます。
   → テストや機能追加が素直になります。

### 3.1 なぜ「フレームワークなし」で作るのか

FastAPI や Flask を使えば数行で書けます。でも本教材では**あえて素手で書きます**。理由は:

- フレームワークは「ルーティング」「JSON 変換」「バリデーション」を隠してくれます。便利な反面、**Claude Code に何を直してもらっているのか**が見えにくい。
- 素のサーバなら、リクエストが来てレスポンスを返すまでの**全部が自分のコード**。だから「ここをリファクタして」「ここで落ちる」が一直線に理解できます。
- 依存ゼロ = 環境差で詰まらない。**学習に集中できます。**

### 3.2 作業ディレクトリを用意する

```bash
# この教材用のディレクトリを作る(リポジトリ内のどこでもOK)
mkdir notes-api
cd notes-api

# Python のバージョン確認(3.10 以上なら何でも可)
python3 --version
```

> この先のコードは `notes-api/app.py` に書いていきます。**コピペでもいいですが、できれば手で写してください。**
> 手で書くと「どこに何があるか」が頭に入り、後のユースケースで Claude Code の変更を**自分で評価できる**ようになります。

## 4. ステップ0:題材コードを自分で書く

ここはユースケースに入る前の「下ごしらえ」です。少しずつ組み立てます。

### 4.1 まず最小:`/health` だけ返すサーバ

最初は「動いた!」を体験するために、ヘルスチェックだけのサーバを書きます。
`notes-api/app.py` を作成し、次を書いてください。

```python
# app.py
# 目的: 標準ライブラリ http.server だけで作る最小の Web サーバ。
#       まずは /health にアクセスすると {"status": "ok"} を返すだけ。

import json
from http.server import BaseHTTPRequestHandler, HTTPServer

# BaseHTTPRequestHandler を継承して「リクエストが来たら何をするか」を書く。
# do_GET という名前のメソッドが、GET リクエストの処理になる(http.server の決まり)。
class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        if self.path == "/health":
            # 1) JSON 文字列を作り、バイト列にエンコードする
            body = json.dumps({"status": "ok"}).encode("utf-8")
            # 2) ステータスコード 200(成功)を返すと宣言
            self.send_response(200)
            # 3) これは JSON ですよ、とヘッダで伝える
            self.send_header("Content-Type", "application/json; charset=utf-8")
            self.end_headers()
            # 4) 本文を書き込む
            self.wfile.write(body)
        else:
            # /health 以外は 404(見つからない)
            self.send_response(404)
            self.end_headers()

def main():
    # ("127.0.0.1", 8000) = 自分のPCの8000番ポートで待ち受ける
    server = HTTPServer(("127.0.0.1", 8000), Handler)
    print("listening on http://127.0.0.1:8000")
    server.serve_forever()  # Ctrl+C で止めるまで動き続ける

if __name__ == "__main__":
    main()
```

起動して確認します。

```bash
# 1つ目のターミナル:サーバを起動
python3 app.py
# -> listening on http://127.0.0.1:8000

# 2つ目のターミナル:アクセスしてみる
curl http://127.0.0.1:8000/health
# -> {"status": "ok"}
```

動いたら成功です。`Ctrl+C` で一度止めてください。

### 4.2 本体:メモの CRUD を足す(あえて伸びしろ付き)

次に、メモを「作る・一覧・取得・削除」できるようにします。
**ここで仕込む“伸びしろ”は後で直すので、今は気にせずそのまま写してください。**
`app.py` を次の内容に**置き換え**ます。

```python
# app.py
# 目的: メモ(Notes)を管理する小さな API。標準ライブラリだけで作る。
#
# 提供するエンドポイント:
#   GET    /health          ヘルスチェック
#   GET    /notes           メモ一覧
#   POST   /notes           メモ作成   body: {"title": "...", "body": "..."}
#   GET    /notes/{id}      メモ1件取得
#   DELETE /notes/{id}      メモ削除
#
# ※ あえて未完成な点(後のユースケースで直す):
#   - 存在しない id を取ると 500 で落ちる(本来は 404)
#   - title が無い POST で 500 で落ちる(本来は 400)
#   - do_GET のルーティングが if/elif の長い分岐で読みにくい

import json
from http.server import BaseHTTPRequestHandler, HTTPServer


# ============================================================
# データ層:メモを保存する箱(今回はメモリ上の辞書。再起動で消える)
# ============================================================
class NotesStore:
    def __init__(self):
        self._notes = {}     # id(int) -> note(dict) の対応表
        self._next_id = 1    # 次に振る id

    def list(self):
        # 値(メモ)だけをリストにして返す
        return list(self._notes.values())

    def create(self, title, body):
        note = {"id": self._next_id, "title": title, "body": body}
        self._notes[self._next_id] = note
        self._next_id += 1
        return note

    def get(self, note_id):
        # ※ 存在しない id だと KeyError が飛ぶ(後でバグとして扱う)
        return self._notes[note_id]

    def delete(self, note_id):
        del self._notes[note_id]


# プロセス全体で1つだけ共有する箱
store = NotesStore()


# ============================================================
# HTTP 層:リクエストを受けて store を呼び、JSON を返す
# ============================================================
class Handler(BaseHTTPRequestHandler):

    # --- 共通のヘルパー:status と payload(dict)を JSON で返す ---
    def _send(self, status, payload):
        body = json.dumps(payload, ensure_ascii=False).encode("utf-8")
        self.send_response(status)
        self.send_header("Content-Type", "application/json; charset=utf-8")
        self.send_header("Content-Length", str(len(body)))
        self.end_headers()
        self.wfile.write(body)

    # --- 共通のヘルパー:リクエスト本文を JSON として読む ---
    def _read_json(self):
        length = int(self.headers.get("Content-Length", 0))
        raw = self.rfile.read(length)
        return json.loads(raw)

    # --- GET の処理 ---
    def do_GET(self):
        if self.path == "/health":
            self._send(200, {"status": "ok"})
        elif self.path == "/notes":
            self._send(200, {"notes": store.list()})
        elif self.path.startswith("/notes/"):
            # "/notes/3" を "/" で割って3番目("3")を取り出し、数値に変換
            note_id = int(self.path.split("/")[2])
            note = store.get(note_id)          # ← id が無いと KeyError → 500
            self._send(200, note)
        else:
            self._send(404, {"error": "not found"})

    # --- POST の処理 ---
    def do_POST(self):
        if self.path == "/notes":
            data = self._read_json()
            # ← "title" や "body" が無いと KeyError → 500
            note = store.create(data["title"], data["body"])
            self._send(201, note)
        else:
            self._send(404, {"error": "not found"})

    # --- DELETE の処理 ---
    def do_DELETE(self):
        if self.path.startswith("/notes/"):
            note_id = int(self.path.split("/")[2])
            store.delete(note_id)
            self._send(200, {"deleted": note_id})
        else:
            self._send(404, {"error": "not found"})


def main():
    server = HTTPServer(("127.0.0.1", 8000), Handler)
    print("listening on http://127.0.0.1:8000")
    server.serve_forever()


if __name__ == "__main__":
    main()
```

## 6. ユースケース②:機能を追加する(build a feature)

> 対応する公式カテゴリ:**Build features and fix bugs**

### 狙い
自然言語で機能を頼み、**複数箇所にまたがる変更**を一度に入れてもらう。今回は
「メモを**更新**する `PUT /notes/{id}` を追加」します。`Handler` に `do_PUT` を足し、`NotesStore` に `update` を足す——という2層の変更が必要です。

### 実際に打つプロンプト

```text
メモを更新する機能を追加して。

- エンドポイント: PUT /notes/{id}
- リクエストボディ: {"title": "...", "body": "..."} (両方とも任意。あるものだけ更新)
- 存在しない id なら 404 を返す
- 成功したら更新後のメモを 200 で返す

NotesStore に update メソッドを足し、Handler に do_PUT を足す方針で。
既存のコードのスタイル(_send / _read_json の使い方)に合わせて。
```

### Claude Code が内部でやること
1. `app.py` を読んで既存スタイルを把握
2. `NotesStore.update(note_id, fields)` を追加(指定キーだけ上書き)
3. `Handler.do_PUT` を追加(パスから id を取り、`_read_json` で本文を読み、`store.update` を呼ぶ)
4. 変更を**差分(diff)として提示**し、あなたの承認を待つ

### あなたが確認すること
- 差分を読む。**「既存スタイルに合っているか」「404 を返す分岐があるか」** をチェック。
- 動作確認:

```bash
# 更新したいメモを1つ作っておく
curl -X POST http://127.0.0.1:8000/notes -H "Content-Type: application/json" \
  -d '{"title": "旧タイトル", "body": "本文"}'

# タイトルだけ更新
curl -X PUT http://127.0.0.1:8000/notes/1 -H "Content-Type: application/json" \
  -d '{"title": "新タイトル"}'
# -> {"id": 1, "title": "新タイトル", "body": "本文"} になればOK
```

### コツ
- **受け入れ条件(仕様)を箇条書きで渡す**と、実装のブレが激減します。「あいまいな日本語1行」より「箇条書き5行」。
- 「**既存のスタイルに合わせて**」は魔法の言葉。命名・エラーの返し方が揃います。
- 大きめの機能なら、先に **Plan モード**(`Shift+Tab` で切替、または `claude --permission-mode plan`)で**編集前に計画だけ**出させ、納得してから実装に進めます。

## 7. ユースケース③:バグを直す(スタックトレースから根本原因へ)

> 対応する公式レシピ:**Fix bugs efficiently**(エラー共有 → 修正案 → 適用)

### 狙い
ステップ0で仕込んだバグ——**存在しない id を取ると 500 で落ちる**——を、エラーメッセージごと渡して直してもらう。
これは現場で最も多い使い方です。「再現コマンド」と「スタックトレース」を渡すのがコツ。

### まずバグを再現する

```bash
# サーバを動かしたまま、存在しない id にアクセス
curl -i http://127.0.0.1:8000/notes/999
```

サーバ側(`python3 app.py` を動かしているターミナル)に、こんなスタックトレースが出ます:

```
Exception occurred during processing of request from ('127.0.0.1', ...)
Traceback (most recent call last):
  ...
  File "app.py", line XX, in do_GET
    note = store.get(note_id)
  File "app.py", line XX, in get
    return self._notes[note_id]
KeyError: 999
```

### 実際に打つプロンプト
**スタックトレースをそのまま貼り付けます。** Claude Code は貼られたトレースから該当行へ一直線に飛べます。

```text
存在しない id でメモを取得すると 500 エラーで落ちる。再現コマンドは
  curl http://127.0.0.1:8000/notes/999
サーバ側のスタックトレースはこれ:

Traceback (most recent call last):
  ...
  File "app.py", line XX, in get
    return self._notes[note_id]
KeyError: 999

原因を特定して、存在しない id のときは 500 ではなく 404 で
{"error": "note not found"} を返すように直して。
ついでに POST で title が無いときも 500 ではなく 400 を返すようにして。
```

### Claude Code が内部でやること
1. トレースの `store.get` → `self._notes[note_id]` を根本原因と特定
2. `NotesStore.get` を「無ければ `None` を返す」形に変えるか、`Handler` 側で存在チェックを足す
3. `do_GET` の `/notes/{id}` 分岐に「`None` なら 404」を追加
4. `do_POST` に「`title` が無ければ 400」を追加
5. 差分を提示

### あなたが確認すること
```bash
# 404 になるか
curl -i http://127.0.0.1:8000/notes/999
# -> HTTP/1.0 404 ... {"error": "note not found"}

# 400 になるか(title を入れずに作成)
curl -i -X POST http://127.0.0.1:8000/notes -H "Content-Type: application/json" -d '{"body": "本文だけ"}'
# -> HTTP/1.0 400 ...
```

### コツ
- **「再現手順 + スタックトレース」をセットで渡す**のが最速。症状だけ伝えるより圧倒的に当たります。
- 「**間欠的に起きる**」「**特定条件でのみ**」など、再現性の情報も添えると精度が上がります。
- 直った後に **「同じ種類のバグが他に無いか探して」** と頼むと、横展開で潰せます(例:`delete` も id が無いと落ちるはず)。

## 8. ユースケース④:レガシーコードをリファクタする

> 対応する公式レシピ:**Refactor code**(古い書き方の検出 → 提案 → 適用 → テスト)

### 狙い
動作は変えずに、**読みやすさ・拡張しやすさ**を上げる。今回の伸びしろは
**`do_GET` / `do_POST` / `do_DELETE` の「長い if/elif でパスを判定」している部分**。
ここを「ルーティングテーブル(パターン → 処理関数の対応表)」に整理します。

### 実際に打つプロンプト(小さく・安全に)

```text
do_GET / do_POST / do_DELETE のルーティングが if/elif の分岐で読みにくい。
"/notes/{id}" のようなパスを正規表現で扱える小さなルーティングの仕組みに
リファクタして。条件:

- 振る舞いは一切変えない(レスポンスもステータスコードも同じ)
- 新しい外部ライブラリは追加しない(標準ライブラリのみ)
- 1ファイル app.py のまま
- まず「変更の方針」を3行で説明してから、差分を出して
```

### Claude Code が内部でやること
- 例えば `(メソッド, パスの正規表現) -> ハンドラ関数` の表を作り、`do_GET` などは「表を引いて該当関数を呼ぶ」だけに簡素化
- 既存の `_send` / `_read_json` は再利用
- **振る舞いを保ったまま**構造だけ入れ替える

### あなたが確認すること
- **差分を読む**:挙動が変わっていないか(特にステータスコード)
- リファクタは**動作確認が命**。章⑤〜⑦で使った `curl` を全部もう一度流して、結果が変わらないことを見る
- 不安なら次章のテストを**先に**作ってからリファクタすると安心(これが理想の順序)

### コツ
- **「振る舞いを変えるな(behavior-preserving)」** を明示する。これがリファクタと「機能変更」の境界線。
- **小さく刻む。** 「全部きれいにして」ではなく「このルーティングだけ」。差分が小さいほどレビューも安全。
- 「**この書き方の何が嬉しいのか説明して**」と聞くと、学びになります(なぜルーティングテーブルが良いのか)。

## 9. ユースケース⑤:テストを自動生成する

> 対応する公式レシピ:**Work with tests** / 公式カテゴリ「後回し作業の自動化」

### 狙い
**追加の依存なし**でテストを書く。Python 標準の `unittest` を使い、`NotesStore` の振る舞いを検証します。
「層を分けておいた」のがここで効きます——HTTP を立てなくても**データ層だけ**テストできます。

### 実際に打つプロンプト

```text
NotesStore のユニットテストを test_app.py に作って。条件:

- 標準ライブラリ unittest を使う(外部ライブラリは入れない)
- 検証する振る舞い:
  - create で id が 1 から連番で振られる
  - list が作った分だけ返す

  - delete 後に get するとちゃんと「無い」扱いになる

- 実行コマンドも教えて
```

> ここで「直した後の仕様に合わせて」と書いているのは、章⑦で `get` の挙動を「無ければ None / 404」に変えたためです。
> もし章⑦をまだやっていなければ、「存在しない id で KeyError になることを確認するテスト」にしてもらえばOK。

### Claude Code が内部でやること
- `app.py` の既存スタイルを読み、`unittest.TestCase` を継承したテストクラスを生成
- `setUp` で新しい `NotesStore()` を用意し、各テストで独立に検証
- **エッジケース**(境界値・異常系)を自分で洗い出して追加

### あなたが確認すること
```bash
# テストを実行(notes-api ディレクトリで)
python3 -m unittest test_app.py -v
# -> すべて ok になればOK
```

- テストが**本当に意味のあるアサーション**になっているか(ただ通すだけの空テストになっていないか)を読む
- **わざとコードを壊して**テストが落ちることを確認すると、テストの信頼性が分かる(例:`_next_id += 1` を消すと連番テストが落ちるはず)

### コツ
- Claude Code は **既存のテストの書き方に合わせて**くれます。最初の1つを作れば、以降は流儀が揃う。
- 「**見落としそうなエッジケースを挙げて**」と頼むと、エラー条件・境界値・想定外入力を提案してくれます。
- これが公式のいう「**書くのを後回しにしがちな作業の自動化**」の中心。テストは最も費用対効果が高いユースケースの1つです。

## 10. ユースケース⑥:依存関係をアップグレードする

> 対応する公式カテゴリ:「後回し作業の自動化(依存更新)」

### 狙い
依存パッケージの更新は地味で面倒、でも放置すると危険。Claude Code に**調査 → 影響範囲の説明 → 更新 → 動作確認**まで任せます。
今回の `app.py` は標準ライブラリのみなので、**依存を1つ足してから**この流れを体験します。

### まず「更新対象」を1つ作る
学習用に、外部ライブラリを1つ導入します(例:`requests`)。古めのバージョンを指定して入れます。

```bash
# notes-api 内で簡易的に(uv を使っている人は uv 環境で)
python3 -m venv .venv
source .venv/bin/activate    # Windows は .venv\Scripts\activate
pip install "requests==2.28.0"
pip freeze > requirements.txt
```

`requirements.txt` に `requests==2.28.0` が記録されます。これが「古い依存」です。

### 実際に打つプロンプト

```text
requirements.txt にある依存を最新の安定版に更新したい。手順をこの順で進めて:

1. requirements.txt の各パッケージについて、現在の固定バージョンと最新版を調べて表にして
2. 破壊的変更(メジャーアップデートの注意点)があれば教えて
3. requirements.txt を最新の安定版に書き換えて
4. 更新後に動かす確認手順(コマンド)を示して

破壊的変更で壊れそうな箇所は、先に警告して。
```

### Claude Code が内部でやること
- `requirements.txt` を読み、各パッケージの最新版・変更点を調査(必要に応じてドキュメント参照)
- **メジャーバージョンの壁**(後方互換が壊れる更新)を洗い出し、影響範囲を説明
- ファイルを書き換え、再インストール・動作確認のコマンドを提示

### あなたが確認すること
```bash
pip install -r requirements.txt   # 更新を反映
python3 -m unittest test_app.py   # テストが通るか(回帰チェック)
```

- **テストがあると依存更新が一気に安全になる**(章⑤の価値がここで出る)
- メジャー更新は一度に1つずつ。まとめて上げると、壊れたとき原因が分からなくなる

### コツ
- 「**まず調査と影響説明だけ。書き換えはこちらの確認後に**」と区切ると、暴走せず安全。
- 公式は「週次の依存監査(weekly dependency audits)」を**スケジュール実行**の例に挙げています(章⑪)。

## 12. ユースケース⑧:CLI でパイプ・スクリプト・CI 自動化

> 対応する公式カテゴリ:**Pipe, script, and automate with the CLI**

### 狙い
Claude Code は **Unix 哲学**に従う1コマンド。対話に入らず `claude -p "..."`(print mode)で**1往復だけ**実行でき、標準入力を渡せます。これで**ログ解析・差分レビュー・CI 組み込み**が自動化できます。

### 例1:ログを流し込んで異常検知

```bash
# サーバのアクセスログ(例)を末尾200行だけ渡して要約させる
tail -200 server.log | claude -p "このログに異常な兆候があれば教えて。無ければ『異常なし』とだけ返して"
```

### 例2:変更ファイルだけセキュリティ観点でレビュー

```bash
# main との差分で変わったファイルだけ渡す
git diff main --name-only | claude -p "これらの変更ファイルをセキュリティ観点でレビューして"
```

### 例3:直近のコミットを要約

```bash
git log --oneline -20 | claude -p "最近のコミットを3行で要約して"
```

### CI に組み込む(イメージ)
公式は GitHub Actions / GitLab CI と組み合わせた**PR レビュー・Issue トリアージの自動化**を挙げています。
発想は「`claude -p` を CI のステップとして実行し、出力をコメントや Slack に流す」だけ。

```yaml
# .github/workflows/review.yml の発想イメージ(疑似コード)
# - PR が開いたら変更差分を claude -p に渡し、レビューコメントを生成して投稿する
```

> 実際の構築は公式の **GitHub Actions / GitLab CI/CD** ガイド、**Non-interactive mode(headless)** を参照。
> ここでは「**`-p` で非対話実行 → 標準入出力で繋ぐ**」という**型**を覚えるのが目的です。

### あなたが確認すること
- `claude -p` は**1往復**で終わる(対話ループに入らない)。CI 向き。
- 自動実行のプロンプトは **「成功とは何か」「結果をどうするか」を明示**する。自律実行なので質問返しができないため。

### コツ
- パイプ(`|`)で**入力を渡し**、`-p` で**1回答を得る**。この2つだけで多くの自動化が組めます。
- `CLAUDE.md` をプロジェクト直下に置くと、**毎セッションの冒頭で読まれる**ので、コーディング規約・レビュー観点を共有できます(公式カテゴリ「指示・スキル・フックでカスタム」)。

## 14. ユースケース⑩:ノートブック→本番パイプライン(データ系)

### まず「探索的な雑コード」を用意(イメージ)


### Claude Code が内部でやること
- 雑コードを「関数分割 + 設定の外出し + 例外処理 + エントリポイント」に整理
- **再現性**(どの環境でも同じ手順で動く)を担保する形に変換

### あなたが確認すること
```bash
NOTES_API_URL=http://127.0.0.1:8000/notes python3 explore.py
# メモが0件でも落ちないか / 環境変数で切り替わるか を確認
```

### コツ
- ノートブック → 本番化の肝は **「関数化・設定の外出し・エラー処理・再現性」** の4点。これを条件に書くと品質が安定。
- 本物の `.ipynb` を渡して「**これをスクリプトに変換して**」も可能。Claude Code はノートブックも読めます。

## 15. ユースケース⑪:スケジュール実行・並列・サブエージェント

> 公式カテゴリ:**Run agent teams** / **Schedule recurring tasks**、公式レシピ:並列(worktree)・サブエージェント

ここは「**1人で逐次やる**」を超える使い方です。題材コードでも考え方は同じなので、**いつ使うか**を中心に。

### 15.1 サブエージェントに調査を委譲する(コンテキストを汚さない)

大きな調査をすると、メインの会話が**ファイルの読み込みでいっぱい**になります。サブエージェントに任せると、**結果だけ**が返ってきます。

```text
サブエージェントを使って、この API のエラーハンドリングが一貫しているか調査して。
各エンドポイントが異常系で何を返すかを表にまとめて報告して。
```

→ サブエージェントが自分のコンテキストでファイルを読み、要約だけを返します。メインの会話はスッキリ保たれます。

### 15.2 並列セッション(worktree)で同時作業

機能追加とバグ修正を**別々のブランチで同時並行**したいとき、worktree を使うと**編集がぶつかりません**。

```bash
# 別のチェックアウト(別ブランチ)を作って、そこで作業
claude --worktree feature-tags     # 例:タグ機能の開発
# 別ターミナルで
claude --worktree fix-validation   # 例:バリデーション修正
```

### 15.3 定期実行(スケジュール)

「毎朝の PR レビュー」「夜間の CI 失敗解析」「週次の依存監査」などを自動化できます。

| 方法 | 動く場所 | 向いている用途 |
|------|---------|--------------|
| **Routines** | Anthropic 管理基盤 | PCを切っても動かしたいタスク。API/GitHub イベント起動も可 |
| **Desktop scheduled tasks** | 自分のマシン | ローカルファイルや未コミット変更に触る必要があるタスク |
| **GitHub Actions** | CI パイプライン | PR 起動や、設定をリポジトリに置きたいタスク |
| **`/loop`** | 起動中の CLI セッション | セッションを開いている間の簡易ポーリング |

```text
# CLI セッション内で簡易ポーリング(例)
/loop 5m 開いている needs-review ラベルの PR を確認して、要約を出して
```

> 自律実行のプロンプトは章⑧と同じく **「成功条件」と「結果の扱い」を明示**します。質問返しができないからです。

### コツ
- **逐次で十分なら逐次で。** 並列・サブエージェントは「独立した作業が複数ある」ときに効く道具です。やみくもに使わない。
- まずは **サブエージェントへの調査委譲**が一番手軽で効果的。「重い調べ物」を投げる癖をつけると、メインの会話が長持ちします。

## 16. まとめ:そのまま使えるプロンプトの型

ユースケースを横断して、**現場でコピペできる型**を抽出しました。`...` を埋めて使ってください。

```text
# 理解する
このコードベースの概要を教えて。何をする〇〇で、ファイル構成と主要な役割は?
〇〇から△△までの処理の流れを、関数を追いながら説明して

# 機能を足す(受け入れ条件を箇条書きで)
〇〇する機能を追加して。
- エンドポイント/関数: ...
- 入力: ...
- 異常系: ...(404/400 など)
- 既存のスタイルに合わせて。まず方針を3行で説明してから差分を出して

# バグを直す(再現コマンド + スタックトレースをそのまま貼る)
〇〇すると落ちる。再現コマンドは ...。スタックトレースはこれ: <貼り付け>
原因を特定して、△△を返すように直して。同種のバグが他に無いかも探して

# リファクタ(振る舞いを変えない)
〇〇が読みにくい。△△に整理して。
条件: 振る舞いを変えない / 新しい依存を足さない / まず方針を説明してから差分

# テスト(依存を増やさない)
〇〇のユニットテストを作って。標準の unittest で。
エッジケース(境界値・異常系)も挙げて足して。実行コマンドも教えて

# 依存更新(調査→確認→適用)
依存を最新安定版に更新したい。まず現在/最新と破壊的変更を表にして。
書き換えはこちらの確認後に。更新後の確認手順も

# コミット/PR
変更を意味のある単位でコミットして(Conventional Commits)。
変更をまとめて PR を作って。説明に「何を・なぜ・どう確認したか」を含めて

# 自動化(非対話・パイプ)
<コマンド> | claude -p "...。結果はこう扱って。無ければ『〇〇』とだけ返して"
```

### 型に共通する4つのコツ
1. **受け入れ条件を箇条書きで。** あいまいな1行より、明確な5行。
2. **「既存のスタイルに合わせて」。** 命名・エラー処理・テストの流儀が揃う。
3. **大きい作業は Plan モードで計画 → 承認 → 実装。** 暴走を防ぐ。
4. **必ず diff を読み、自分で動作確認する。** 最終責任は人間。

## 17. 安全に使うための注意点

ユースケースを実務に持ち込む前に、最低限これだけは。

- **変更は必ずレビューする。** Claude Code は diff を提示します。**読まずに承認しない。** 特にインフラ(章⑨)とコミット/PR(章⑦)。
- **破壊的操作は段階を区切る。** 「まず調査だけ」「書き換えは確認後に」と言えば、勝手に進みません(章⑩⑥)。
- **依存更新・リファクタはテストとセットで。** 章⑤のテストがあると、回帰を即座に検知できます。順序の理想は「テスト → リファクタ/更新」。
- **外部送信を含む操作に注意。** PR 作成・Slack 投稿・デプロイは「外向き」の操作。**一度公開すると取り消せない**ことを意識して。
- **自動/スケジュール実行は成功条件を明示。** 自律実行は質問を返せません。「何が成功か」「結果をどうするか」を必ず書く。
- **`CLAUDE.md` に規約を置く。** チームの約束事(命名・テスト方針・禁止事項)を書いておくと、毎回指示しなくても守られます。

## 18. この教材のまとめと次の一歩

### やったこと
- 公式「What you can do」「Common workflows」と実例記事を読み解き、**Claude Code のユースケース全体像**を地図にした
- **テンプレート・フレームワークを使わず**、標準ライブラリだけで Notes API を**自作**した
- その同じコードに対して、**理解 → 機能追加 → バグ修正 → リファクタ → テスト → 依存更新 → コミット/PR** という**開発サイクル一周**を Claude Code で回した
- **自動化(パイプ/CI/インフラ)・データパイプライン化・並列/サブエージェント/スケジュール**の使いどころを押さえた
- **再利用できるプロンプトの型**を手に入れた

### 一番大事な気づき
ユースケースは数あれど、骨格は同じ——**「理解させる → 条件を箇条書きで頼む → diff を読む → 動作確認する」**。
この4拍子を、**自分の実プロジェクト**でも回してください。題材が変わっても、型は同じです。

### 次の一歩(おすすめの順)
1. **自分の手元のリポジトリ**で `claude` を起動し、章⑤の「コードベースを理解する」から試す
2. 小さなバグや「テストが無い関数」を1つ選び、章③・章⑤を適用する
3. `CLAUDE.md` をプロジェクトに置き、チームの規約を書く(章⑧)
4. うまくいったワークフローを**スキル**化して再利用する(教材02・公式 Skills 参照)
